# Watchlist Architecture: 3-Tier Data Strategy

This document formally defines the physical home and lifecycle for every piece of data in the system, respecting the Render free-tier constraints and the complexity of managing 50+ stocks requiring 1-year historical OHLCV data.

## User Review Required
> [!IMPORTANT]
> Please review the exact categorization of data into the three levels below. Pay special attention to the **Redis Keyspaces** section to ensure the TTLs match the business requirements.

---

## Architectural Design: The 3 Levels

### Level 1: Live Data (Local JVM Memory)
**Rule: Never persists. Calculated & served on the fly.**
*   **Data:** Last Traded Price (LTP), Today's Change %, Live Volume.
*   **Storage:** Local Caffeine Cache inside the Service process.
*   **Lifecycle:** TTL of 5 to 15 seconds. If 100 users refresh the page within 5 seconds, only 1 live API call goes to Kite.
*   **Restart Behavior:** Completely wiped every 15 mins (Render restart). Repopulates instantly upon the first user request.

### Level 2: Redis Cache (Distributed In-Memory)
**Rule: Handles all heavy computation and cross-process data sharing.**
We categorize Redis data into four distinct groups based on your constraints:

1. **Data cached for a complete day (TTL: 24h)**
   *   *What:* Computed daily indicators (SMA 50/200, MACD, RSI 14, Drawdown %).
   *   *Source:* Generated by the 9:15 AM Cron Job.

2. **Data computed over multiple days (TTL: 7 Days)**
   *   *What:* Weekly or Monthly aggregated statistics (e.g., Sector rotation strength, Max 1-Year Drawdown). 
   *   *Source:* Slow-drifting data that doesn't need daily recalculation.

3. **Data relying SOLELY on Redis with NO Postgres Fallback (TTL: 2 Days)**
   *   *What:* **1-Year Historical OHLCV Daily Candles for 50+ stocks.**
   *   *Why:* Storing 18,250+ (50 * 365) JSON objects in Postgres every day bloats the database unnecessarily.
   *   *Recovery:* If Redis dies, the API Service *fails fast* for history requests and queues an asynchronous command for the Cron Job to aggressively re-fetch the OHLCV data from Kite API.

4. **Data that refreshes every 2 to 15 minutes (TTL: 15m)**
   *   *What:* Intraday aggregations (e.g., "Is the index strongly trending in the last 15 min?"). This provides fast access to short-term trends.

### Level 3: PostgreSQL (Persistent Database)
**Rule: Only stores structural configuration and lightweight snapshots.**
*   **Data:** 
    1. The Master Stock list and Tag configuration.
    2. A daily `JSONB` snapshot of the **Computed Indicators** (from Level 2.1).
*   **Why no OHLCV data here?** Postgres is strictly a fallback for the *lightweight* indicators so that the Watchlist Dashboard continues to load under 50ms even if Redis dies. The heavy OHLCV data is explicitly kept out to save costs and maintain database health.

## Integration Flow

1. **At 9:15 AM:**
   * Cron Job reads Stock list from Postgres.
   * Fetches 1-Year OHLCV from Kite → Stores into **Redis (No PG Backup)**.
   * Runs `ta4j` on OHLCV to output Indicators (SMA, RSI, etc.).
   * Stores Indicators into **Redis (24h TTL)** AND **Postgres (`JSONB` snapshot)**.
2. **On User API Request:**
   * Service requests indicators from **Redis**.
   * If Redis is dead → Fallback to **Postgres JSONB snapshot**.
   * Service fetches **Live LTP** from Kite (held in 10-second Caffeine cache).
   * Merges Live LTP + Indicators = Returns JSON to Frontend.
