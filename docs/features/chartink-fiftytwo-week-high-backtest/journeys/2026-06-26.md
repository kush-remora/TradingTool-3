# Chartink 52-Week High Backtest

## Why This Was Built

Kush needed a reusable backtest job for a Chartink-exported 52-week-high screener file stored in `manual-input/`. The immediate goal was not app UI behavior; it was a large, AI-friendly trade dataset that can help compare safer patterns across market-cap buckets, sectors, and exit strategies.

## What Was Implemented

- Added a CSV reader for `date`, `symbol`, `marketcapname`, and `sector` signal rows.
- Added a pure Kotlin backtest engine that:
  - buys at the next trading day's open
  - supports fixed target and stop-loss exits
  - records holding days, return, MFE/MAE, and forward returns
  - distinguishes `NO_PRICE_DATA` from `NO_NEXT_TRADING_DAY`
- Added a thin service that loads candles by symbol from the repo's Supabase-backed `daily_candles` table.
- Added a cron job:
  - input: `manual-input/Backtest 52 week high first time.csv`
  - output: `build/reports/chartink-fiftytwo-week-high-backtest/`
- Added focused engine tests for target-hit, conservative ambiguous-bar handling, and market-cap summary grouping.

## Key Decisions And Tradeoffs

- Used a pure engine plus thin job structure to keep logic testable and avoid stuffing research logic into `main()`.
- Used daily candle data already present in Supabase instead of external scraping or Kite token resolution.
- Chose conservative same-day ambiguity handling: if both stop and target are touched in one daily bar, the engine assumes stop-loss first.
- Shipped two default strategies instead of one:
  - `target_5_stop_5`
  - `target_10_stop_5`
- Kept the output trade-level and flat so downstream AI analysis stays simple.

- If this workflow becomes routine, expose the backtest config through a small resource or config file instead of keeping strategy values only in the job.
