# Delivery Breakout Scanner V1

## Feature and purpose

Built a delivery-first validation scanner so Kush can review the full `stock_delivery_daily` universe and check whether unusual absolute delivery participation is surfacing quiet pre-breakout names before the move becomes obvious.

## Implemented

- Added a new backend scanner package under `core/src/main/kotlin/com/tradingtool/core/strategy/deliverybreakout/`.
- Added `GET /api/strategy/delivery-breakout/dashboard`.
- Added Stage 1 filtering that keeps only stocks where:
  - current-day volume is at least `10,000`
  - current-day volume is above the prior 10-trading-day max volume
  - current-day delivery quantity is above the prior 10-trading-day max delivery quantity
- Added Stage 2 enrichment that computes:
  - current close
  - current close % change
  - quiet clue on `D-1` or `D-2`
  - breakout-today flag
  - `SMA200`
  - `% from SMA200`
  - near-`SMA200` status
- Added a new React console page and route: `Delivery Breakout`.
- Added focused Kotlin analyzer tests and a frontend page render test.

## Key decisions and tradeoffs

- Kept the backend scan delivery-first and only enriched shortlisted symbols with candle data to keep the logic readable and cheaper than a price-first full-universe pass.
- Kept `delivery %` as context only. It is returned for review but does not block v1 inclusion.
- Used a simple `> 2%` split for the breakout-today flag so the table can show both still-quiet and already-confirmed names.
- Kept the buy-zone interpretation informational only. Stocks below `SMA200` are still shown because the feature’s first job is validation of the delivery pattern, not filtering away uncomfortable names too early.
- Kept the UI as a ranked table only. No date picker or advanced filters were added in v1 because the immediate goal is daily pattern validation.
