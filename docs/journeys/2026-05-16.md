# Bollinger Backtest v1 Journey (2026-05-16)

## Feature and Why
Implemented Bollinger backtest with explainable trade-level breakdown so each entry/exit can be audited, not just final P&L.

## What Was Implemented
- New backend Bollinger backtest request/response models.
- New stateless backtest engine with config-driven rules.
- New API endpoint: `POST /api/strategy/bollinger/backtest`.
- New frontend Bollinger Backtest page under Screener tab.
- Summary cards + trade table + expandable criteria snapshots for entry/exit.
- Time period config via `lookbackMonths`.

## Key Decisions and Tradeoffs
- Universe is runtime UI input (not persisted in config file).
- Daily-candle simulation for v1 simplicity and readability.
- Exit mode is explicit toggle (`MIDDLE_BAND` or `OVERBOUGHT`).
- Drilldown depth kept at trade snapshot level; no daily timeline yet.

## Validation Run and Outcomes
- Backend compile: `mvn -q -DskipTests compile` passed.
- Frontend build: `npm --prefix frontend run -s build` passed.
- Frontend test: `npm --prefix frontend run -s test:run -- src/pages/BollingerBacktestPage.test.tsx` passed.

## Next Follow-ups
1. Add backend unit tests for exit-mode behavior and SL/TP/max-hold precedence.
2. Add optional daily trade path timeline in drilldown.
3. Add API-level validation helper for malformed backtest config.
