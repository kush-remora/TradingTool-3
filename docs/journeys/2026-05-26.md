## Feature
Stock Instrument Token Audit Script

## Why It Was Built
Investigate repeated Kite failures by verifying whether watchlist `instrument_token` values drifted from current Kite instrument mapping.

## What Was Implemented
- Added a standalone audit job: `StockInstrumentTokenAuditJob`.
- Loads all `stocks` rows, filters NSE, and fetches Kite instruments.
- Compares DB token vs Kite token by `exchange + tradingsymbol`.
- Classifies each stock as `MATCHED`, `MISMATCH`, or `MISSING_IN_KITE`.
- Writes reports to:
  - `build/reports/stock-instrument-token-audit/latest.json`
  - `build/reports/stock-instrument-token-audit/latest.csv`

## Key Decisions / Tradeoffs
- Read-only audit only; no automatic DB updates to avoid accidental remaps.
- Kept matching simple and explicit on NSE exchange and symbol key.
- Reused existing Kite auth flow from token table for consistency.

## Validation Run and Outcomes
- Ran: `mvn -q -pl cron-job -am -DskipTests compile`
- Outcome: Passed.

## Next Follow-ups
- Optionally add a second script to auto-generate SQL `UPDATE` statements only for `MISMATCH` rows (manual apply).

## Feature
Centralized Instrument Token Resolver + Audit Auto-Correction

## Why It Was Built
Token resolution drift and symbol-suffix edge cases were repeatedly breaking workflows. The same fallback logic had to be reimplemented in multiple places, causing month-long operational churn.

## What Was Implemented
- Added shared runtime resolver: `InstrumentTokenResolverService`.
  - Exact match first (`NSE:SYMBOL`).
  - Fallback second (`NSE:SYMBOL-BE`).
  - Emits detailed resolution diagnostics (expected keys, matched key, candidate keys).
- Routed existing resolver wrappers (`index constituents`, `groww watchlist`) through this shared service.
- Updated write-path behavior:
  - `IndexConstituentSyncService` now pre-validates token resolution and fails before writes if unresolved symbols exist.
  - `DeliveryReconciliationService` now fails before writes if unresolved symbols exist.
  - Failure messages include symbols + expected keys + candidate keys for Telegram failure alerts.
- Extended DAO layer for correction operations:
  - `stocks` token update by `(symbol, exchange)`.
  - `index_constituents` token update by active `symbol`.
- Rebuilt `StockInstrumentTokenAuditJob`:
  - default check mode,
  - `--apply` auto-correction mode,
  - updates both `stocks` and `index_constituents`,
  - report output contains mismatch diagnostics and correction counts.
- Added index sync Telegram success summary payload with totals and unresolved count.

## Key Decisions / Tradeoffs
- Resolver fallback is intentionally narrow (`-BE` only) to avoid speculative matching.
- Fail-fast policy is enforced before write operations to prevent partial invalid writes.
- Auto-correction is intentionally manual-triggered (`--apply`) to keep control explicit.

## Validation Run and Outcomes
- `mvn -q -pl core,cron-job -am -DskipTests compile` passed.
- `mvn -q -pl core test -Dtest=IndexConstituentSyncServiceTest` passed.
- Audit run (check mode) succeeded with zero unresolved symbols and produced artifacts.
