# Feature Journey - NSE `-IV` Instrument Token Fallback

## Feature Name and Why It Was Built
- Feature: Instrument token resolver support for NSE symbols where Kite uses `-IV` suffix.
- Why: Symbols like `PGINVIT` were not resolving because Kite stores the tradingsymbol as `PGINVIT-IV`.

## What Was Implemented
- Extended `InstrumentTokenResolverService` NSE fallback logic to try a suffix list instead of only `-BE`.
- Added `-IV` to the NSE fallback suffix list.
- Updated expected-key diagnostics to include all configured NSE fallback keys.
- Added unit test coverage for `PGINVIT -> PGINVIT-IV -> 894465`.

## Key Decisions and Tradeoffs
- Chose a minimal list-based fallback (`-BE`, `-IV`) to keep behavior explicit and readable.
- Avoided broader fuzzy matching so we do not accidentally resolve to incorrect NSE variants.

## Validation Run and Outcomes
- Ran: `mvn -pl core -Dtest=KiteInstrumentTokenResolverTest test`
- Outcome: Passed.

## Next Follow-ups
- If more consistent NSE suffix variants appear in production logs, add them explicitly to the fallback list with tests.

# Feature Journey - Kite Startup Token Validation

## Feature Name and Why It Was Built
- Feature: validate the persisted Kite access token before starting market services.
- Why: the backend was accepting an expired `kite_tokens` row as authenticated, then `KiteTicker` hit a `403 Forbidden` WebSocket handshake instead of failing early with a clear refresh instruction.

## What Was Implemented
- Added startup-time Kite session validation right after loading the latest token from the database.
- Normalized persisted access tokens with `trim()` before applying them to the SDK.
- Added a small startup helper so the stale-token path can be regression-tested without a live Kite call.

## Key Decisions and Tradeoffs
- Decision: validate at startup rather than only catching ticker failures later, because the token table is the auth source of truth.
- Tradeoff: startup now performs one extra authenticated Kite API call, but it avoids booting into a half-authenticated state.

## Validation Run and Outcomes
- `mvn -pl core -Dtest=KiteConnectClientTest test` passed.
- `mvn -q -pl service -Dtest=KiteStartupTokenValidationTest,ApplicationTest test` is currently blocked by pre-existing `service` compile errors around `HotSmaScannerService` wiring in `ServiceModule.kt`.

## Next Follow-ups
- If token refresh reliability becomes noisy, consider surfacing the validation failure directly in a health/admin endpoint too.
