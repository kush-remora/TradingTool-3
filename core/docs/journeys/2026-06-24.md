# Hot SMA Multi-Universe Select

## Feature and why it was built

Added multi-select support to the `SMA Buy Zone Screener` universe dropdown so one scan can cover several existing universes at once instead of forcing separate runs.

## What was implemented

- Changed the Hot SMA request contract from `indexKey` to `indexKeys` across frontend types, resource validation, and core models.
- Updated the React page to use Ant `Select` in `multiple` mode and persist the selected universe list in local storage.
- Updated Hot SMA resource validation to normalize, de-duplicate, and require at least one selected universe key.
- Updated the Hot SMA scanner service to resolve and merge members from multiple universes into one result set while keeping each row's original `indexKey`.
- Updated focused frontend and resource tests to cover the multi-select request flow.

## Key decisions and tradeoffs

- Kept the result table and scanner logic unchanged beyond universe selection so the feature stays small and readable.
- Preserved row-level `indexKey` from the source membership instead of replacing it with one combined label, which keeps mixed-universe output explicit.
- Defaulted to the first available universe only when no persisted selection exists, matching the previous page behavior as closely as possible.

## Validation run and outcomes

- `npm --prefix frontend run test:run -- HotSmaPage.test.tsx` passed.
- `mvn -q -pl resources -am test -Dtest=HotSmaRequestValidationTest -Dsurefire.failIfNoSpecifiedTests=false -Dkotlin.compiler.execution.strategy=in-process` passed.
- `mvn -q -pl core -am test -Dtest=HotSmaScannerServiceTest -Dsurefire.failIfNoSpecifiedTests=false -Dkotlin.compiler.execution.strategy=in-process` passed.
- `npm --prefix frontend run build` passed.
- A separate first Maven attempt hit a Kotlin daemon crash in `resources`; rerunning in-process completed cleanly.

## Next follow-ups

- If mixed-universe scans become the default workflow, consider surfacing the selected universe count or names near the summary line.
- If the same symbol appearing in multiple universes ever needs to retain all memberships, revisit the current symbol-level de-duplication in the scanner service.

## Shared live market snapshot contract

### Feature and why it was built

Reworked the `Live Market` path so all screeners can use one shared component with the same behavior in both live and non-live states instead of depending on page-specific fallback wiring.

### What was implemented

- Extended `/api/stocks/quotes` to return `change_percent` alongside `ltp`, `open`, `high`, `low`, and `volume`.
- Updated `useStockQuotes` so it still fetches the latest snapshot after market hours instead of going blank when live ticks are unavailable.
- Updated `LiveMarketWidget` to merge live SSE fields with snapshot fields and show O/H/L/V details in both modes.
- Updated `renderLiveMarketCell` to accept the shared snapshot object and added a shared `% change` resolver.
- Wired `HotSmaPage`, `PhaseDScannerPage`, `DeliveryBreakoutScannerPage`, `VolumeShockerDashboardPage`, `WyckoffPhase1Page`, and `TradeJournalTable` onto the same snapshot-backed widget path.
- Standardized `Live Market` sorting on those pages to use `% change` rather than raw price wherever that column is present.

### Key decisions and tradeoffs

- Chose the existing `/api/stocks/quotes` route as the fallback data source instead of extending every screener response with duplicate market snapshot fields.
- Kept live SSE as the preferred source, but let snapshot fields fill any missing gaps so the visible UI stays stable before or after live ticks.
- Left some older screener-specific fields in place where they already existed, but the shared widget no longer depends on them for the main fallback behavior.

### Next follow-ups

## Shared Wake-Up volume column

### Feature and why it was built

Added a shared live/non-live volume-shock column so `SMA Buy Zone` can rank crowded `SMA200` candidates by participation quality using the same Wake-Up logic already proven useful on Phase D review.

### What was implemented

- Added `previous_day_volume` to the shared `/api/stocks/quotes` snapshot response.
- Extracted the Phase D wake-up logic into a shared `WakeUpVolumeCell` component plus shared helper functions.
- Kept `PhaseDScannerPage` on the same wake-up behavior through that shared component instead of page-local logic.
- Added a `Wake-Up` column to `HotSmaPage` using the same shared component and quote snapshot source.
- Standardized wake-up sorting on current volume vs `T-1` volume ratio using the shared helper path.

### Key decisions and tradeoffs

- Reused the quote snapshot API rather than adding more screener-specific backend columns just for Wake-Up.
- Compared live intraday volume directly against `T-1` volume, which matches Kush’s stated decision rule more closely than the earlier page-local fallback behavior.
- Kept the cell dense but readable by showing a tag plus `Now/Day`, `T-1`, and volume-difference lines directly in the table cell.
