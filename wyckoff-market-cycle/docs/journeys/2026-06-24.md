# Phase D Scanner UI Redesign

## Feature and why it was built

Redesigned the `Phase D Ignition Scanner` screen so it better matches the real workflow already implemented in the product. The previous page was functional but visually flat and spreadsheet-heavy, which made it harder to understand that the current screen is a Phase C intake and review layer rather than a true live ignition engine.

## What was implemented

- Reworked the page header into a clearer context block that explains what the screen currently does.
- Added compact watchlist health cards for total candidates, resolved tokens, unresolved tokens, and freshly seen rows.
- Redesigned the upload area into a more intentional import panel with refresh and import feedback.
- Added local search and quick filters for `all`, `resolved`, and `unresolved` candidates.
- Reorganized the table into grouped columns:
  - candidate identity
  - status and token resolution
  - dry-up profile
  - structural context
  - ownership and quality context
- Added a focused frontend render test for the redesigned screen.

## Key decisions and tradeoffs

- Kept the backend contract unchanged and limited the work to frontend structure and clarity.
- Chose grouped contextual cells over many raw columns to reduce scanning fatigue while preserving the same information.
- Kept the page honest about its current behavior instead of pretending the Phase D engine already exists.
- Avoided adding advanced controls or new actions because the current user need is still import-plus-review, not execution from this screen.

## Validation run and outcomes

- `npm --prefix frontend run test:run -- PhaseDScannerPage.test.tsx` passed.
- `npm --prefix frontend run build` passed.
- Build still reports the existing Vite large-chunk warning; no new build failure was introduced.

## Next follow-ups

- When the actual Phase D trigger engine exists, replace the explanatory "current screen purpose" panel with live monitoring status and alert outcomes.
- Consider adding sortable columns or a compact pin/priority signal if the watchlist becomes materially larger.

## Follow-up correction

- Replaced the Tailwind-style utility-class version of the page with an Ant Design-only implementation after visual verification showed the frontend was rendering those classes as near-raw HTML.
- The corrected screen now relies only on components and styling already active in this app, which is the safer and more consistent path for this repo.

## CSV import correction

- Investigated why many uploaded Phase C fields were landing as `null` in the database after the CSV headers were cleaned up.
- Root cause was twofold:
  - the frontend parser still searched only for old header names
  - the parser used plain comma splitting, which broke quoted CSV values such as `3,410.2`
- Fixed the parser to:
  - support both old and cleaned header aliases
  - parse quoted CSV rows safely before mapping fields
- Added a focused test that covers the cleaned header version of the CSV and verifies the key fields survive mapping.

## Schema naming correction

- Renamed the `phase_c_watchlist` schema vocabulary to match the cleaned CSV naming rather than keeping legacy placeholders.
- This affects database columns, Kotlin DTO/model property names, DAO bindings, table definitions, and frontend payload field names.
- Added a dedicated migration with explicit `ALTER TABLE ... RENAME COLUMN ...` commands so existing data can be preserved while the naming is cleaned up.

## Net profit naming correction

- Replaced the misleading `net_profit_3q_ago` naming with `net_profit_after_tax` across the frontend parser, Kotlin DTO/model/DAO path, and schema definitions.
- Kept CSV parsing backward-compatible by accepting both `net_profit_after_tax` and the older `net_profit_3q_ago` header during import.
- This aligns the codebase with the clarified meaning of the column: current quarterly PAT rather than a three-quarters-ago historical value.

## Phase 2 delivery validation

### Feature and why it was built

Built the next funnel layer after the existing Phase 1 shortlist so the current watchlist can answer a more Wyckoff-specific question: during the quiet setup, did meaningful delivery conviction actually show up?

### What was implemented

- Added Phase 2 current-state fields to `phase_c_watchlist` for delivery status, reason, evaluation date, latest delivery metrics, and 10d/20d conviction counts.
- Added a Kotlin Phase 2 delivery analyzer that:
  - loads recent delivery and candle history
  - builds a basement-price delivery baseline
  - checks delivery spike plus supportive delivery %
  - assigns `PASSED`, `WATCH`, `NOT_PASSED`, or `DATA_MISSING`
- Added `POST /api/strategy/phase-c/delivery-validation/run` so the same watchlist can be re-evaluated on demand without a fresh CSV upload.
- Updated the Phase D page into a clearer Phase 1 plus Phase 2 workflow with:
  - a `Run Delivery Validation` button
  - summary cards for Phase 2 results
  - `All Phase 1 Stocks` and `Delivery Conviction` tabs
  - shared columns that show both structural context and delivery-validation fields
- Added focused backend analyzer tests and expanded frontend page tests for the new run flow and tabs.

### Key decisions and tradeoffs

- Kept the table as a current-state watchlist and wrote the latest Phase 2 result back onto the same symbol row instead of introducing a history table yet.
- Used one shared threshold set across all caps for the first cut to keep the rule easy to inspect and tune.
- Kept the implementation count-based rather than score-based so the logic stays readable.
- Used both delivery quantity and delivery % in the Phase 2 rule, while leaving the earlier quiet-range logic in Phase 1 where it already belongs.

## Phase C fresh-field refresh

### Feature and why it was built

Stopped treating uploaded Phase C market fields as permanently fresh. The uploaded CSV is still useful as a seed snapshot, but fields such as close, percent change, volume, 52-week range, and distance-from-range change with the market and should be refreshable from daily candles.

### What was implemented

- Added `market_fields_updated_on` to `phase_c_watchlist`.
- Kept upload behavior simple:
  - uploaded market fields are still stored
  - every insert and re-upload resets `market_fields_updated_on` to `NULL`
- Added a dedicated backend refresh flow:
  - `POST /api/strategy/phase-c/fresh-fields/update`
  - syncs daily candles from Kite for the full watchlist
  - recalculates only the volatile market fields
  - updates all rows in one write step
- Added a pure Kotlin calculator for:
  - latest close
  - latest day percent change
  - latest volume
  - 52-week high and low
  - `dist_200d_high_pct`
  - `dist_200d_low_pct` using the 52-week low while keeping the current column name
- Updated the Phase D page with:
  - an `Update Fresh Fields` button
  - a freshness banner above the table
  - refresh success feedback
  - row-level visibility for whether fresh fields are pending or updated

### Key decisions and tradeoffs

- Kept `phase_c_watchlist` as the single current-state table to avoid introducing history-table complexity right now.
- Chose manual refresh instead of auto-refresh on dashboard load so market sync remains explicit and user-controlled.
- Made the refresh all-or-nothing so a mixed stale/updated watchlist cannot be written accidentally.
- Left the existing `dist_200d_low_pct` name in place for now to avoid another rename cycle, while correcting its calculation semantics.

### Validation run and outcomes

- `mvn -q -pl core -am test -Dtest=PhaseCFreshFieldCalculatorTest -Dsurefire.failIfNoSpecifiedTests=false` passed.
- `mvn -q -pl service -am -DskipTests compile` passed.
- `npm --prefix frontend run test:run -- PhaseDScannerPage.test.tsx` passed.
- `npm --prefix frontend run build` passed.
- Frontend test output still shows the existing jsdom Ant stylesheet parsing warnings; the new feature did not introduce a functional failure.

### Next follow-ups

- Add backend workflow/integration tests around refresh rollback if the repo test harness gets a simple way to exercise JDBI plus Kite sync boundaries.
- Consider surfacing the latest `market_fields_updated_on` in the top summary cards if the watchlist grows enough that freshness becomes the primary operational check.

## Wake-Up signal clarity

### Feature and why it was built

Simplified the `Wake-Up` column on the Phase D review table so the live-volume cue is easier to read during manual dashboard scans. The earlier `5x` threshold and generic badge wording hid useful intermediate cases that Kush still wants to notice, especially when live volume is already meaningfully above the prior day but not yet extreme.

### What was implemented

- Lowered the live-volume trigger from `5x` to `2x` versus `T-1` volume.
- Changed the badge copy to show the actual live-volume multiple, for example `VOL 2.0x` or `MOVE + 2.5x`.
- Kept the price-move signal at `4%`.
- Expanded the cell details to show both the current live volume and the `T-1` baseline volume under the badge.
- Hardened the shared live market hook to use safe global timer APIs so delayed teardown callbacks do not crash focused frontend tests after jsdom cleanup.

### Key decisions and tradeoffs

- Kept the change frontend-only because the signal is derived from already available live market data plus the row's stored `T-1` volume.
- Chose direct ratio labels over more categories so the dashboard tells the truth without extra interpretation.
- Left the existing live recalculation behavior untouched; only the threshold and presentation changed.

### Validation run and outcomes

- `npm --prefix frontend run test:run -- PhaseDScannerPage.test.tsx` passed.

### Next follow-ups

- If the column still feels noisy in practice, consider adding a muted non-trigger ratio display for `1.0x` to `1.9x` without changing the badge status.

The live-price path was degrading app performance because the backend subscribed to the Kite ticker at startup for the full active universe, then kept growing that subscription set as pages reconnected. The SSE endpoint also used one polling thread per client and the frontend still tried to open live connections outside market hours.

### What was implemented

- Added a small NSE market-hours helper on backend and frontend, including weekend blocking.
- Removed startup-wide live ticker subscription from `Application.kt`; the ticker now starts lazily only when a page actually requests symbols.
- Refactored `KiteTickerService` to keep per-token reference counts, subscribe on first consumer, unsubscribe on last consumer, and disconnect when no live pages remain.
- Reworked `/api/market/live` to:
  - reject live streams when market hours or auth are unavailable
  - stream updates from `TickStore` callbacks instead of a polling loop
  - clean up SSE sessions and release tokens on disconnect/send failure
- Updated frontend live hooks so:
  - SSE does not open outside market hours
  - page unload/unmount closes the connection
  - quote polling avoids live requests when the market is closed

### Key decisions and tradeoffs

- Kept the implementation intentionally simple and single-user oriented instead of adding broker-style shared streaming infrastructure.
- Left the live payload shape unchanged so current widgets keep working without a separate UI migration.
- Kept the existing Kite `modeFull` subscription for now to avoid an unnecessary data-contract change during the lifecycle fix.

- A broader `PhaseDScannerPage.test.tsx` run still has one pre-existing timeout in the refresh-error case; the targeted live-market checks and build remained green.

### Next follow-ups
