# Wyckoff Index Sync Universe Trim

## Why this was built
The index constituent sync config still contained overlapping baskets (`NIFTY 100`, `NIFTY MIDCAP 100`, `NIFTY SMALLCAP 100`, `NIFTY 500`). We only want three cleaner, non-overlapping coverage buckets for this module.

## What was implemented
1. Replaced the existing index list in `wyckoff-market-cycle/config/index_sync_config.json`.
2. Kept only:
   - `Nifty Smallcap 500`
   - `NIFTY LargeMidcap 250`
   - `NIFTY MICROCAP 250`
3. Removed the broader overlapping feeds so config alone now defines the intended sync universe.

## Key decisions and tradeoffs
- Chose a config-only change because there is no dedicated overlap-resolution logic in this module today.
- Kept the existing `batchSize` unchanged to avoid unrelated behavior changes.
- Used explicit index-specific CSV feeds instead of keeping a broader basket plus downstream dedupe logic.

## Validation run and outcomes
- Pending: format and structure validation with `jq . wyckoff-market-cycle/config/index_sync_config.json`
- Pending: diff review after config update

## Next follow-ups
- If future indices are added, keep them non-overlapping unless the sync path gains explicit dedupe rules.

## Index Sync Smallcap 500 CSV Fix

### Why this was built
The `nifty_smallcap_500` sync started failing because the configured constituent URL returned an HTML page instead of a CSV payload, which surfaced downstream as a misleading missing-symbol-column parse error.

### What was implemented
1. Updated `wyckoff-market-cycle/config/index_sync_config.json` to use the live `Nifty Smallcap 500` CSV feed:
   - from `ind_niftysmallcap500list.csv`
   - to `ind_niftysmallcap500_list.csv`
2. Hardened `core/src/main/kotlin/com/tradingtool/core/indexconstituents/IndexConstituentCsvSource.kt` to explain whether the source returned HTML, an empty body, or a non-matching header row.
3. Added focused parser tests for both HTML-body and missing-header diagnostics.

### Key decisions and tradeoffs
- Fixed the source URL in config instead of adding fallback URL guessing in code.
- Kept the parser behavior strict; it still rejects non-CSV bodies, but now the error message points to the real upstream problem.
- Limited diagnostics to the first non-empty line so failures stay readable in cron logs.

### Validation run and outcomes
- `mvn -q -pl core -Dtest=IndexConstituentCsvSourceTest test`:
  - passed (`core/target/surefire-reports/com.tradingtool.core.indexconstituents.IndexConstituentCsvSourceTest.txt`)
- Live URL probe confirmed:
  - `ind_niftysmallcap500list.csv` returns `text/html`
  - `ind_niftysmallcap500_list.csv` returns the expected CSV header with `Symbol`

### Next follow-ups
- If other Nifty feeds migrate to `_list.csv`, update config entries directly rather than masking broken sources in parser logic.
