# Feature Journey - Wyckoff Phase-1 Scanner (Universe-Only Selection)

## Feature Name and Why It Was Built
- Feature: remove symbol-source path from Wyckoff Phase-1 scanner controls.
- Why: scanner was defaulting to all watchlist symbols (~128), causing runs to ignore the intended universe-key-only flow and creating incorrect default scope.

## What Was Implemented
- Removed `Symbol Source (optional)` controls from `WyckoffPhase1Page`.
- Removed watchlist symbol fetch/state from the page.
- Updated scanner request build to send only selected `universeKeys` and strict-base-filter flag.
- Simplified local storage filter model for this page to:
  - `universeKeys`
  - `strictBaseFilter`
- Updated page tests for universe-only behavior and removal of symbol-source UI.
- Removed unused frontend type `WyckoffPhase1SymbolSourceMode`.

## Key Decisions and Tradeoffs
- Decision: keep backend request contract unchanged (`symbols` remains optional), but stop using it from this page.
- Tradeoff: symbol-only debug runs are no longer possible from this scanner UI, in favor of predictable universe-key behavior aligned with Phase-1 intent.
- Decision: preserve strict filter toggle behavior and persistence unchanged.

## Validation Run and Outcomes
- `npm --prefix frontend run test:run -- WyckoffPhase1Page` passed (4/4).
- `npm --prefix frontend run build` passed.
- Kotlin diff gate check run: no `.kt`/`.kts` files changed.

## Next Follow-ups
- Consider backend-side telemetry of `universeKeys` per run for auditability.
- Optionally remove now-unused symbol normalization in related non-Phase-1 screens if not needed.

## Rule revisit - markdown absorption alignment

- Feature: relax Phase-1 thresholding to better admit weak-price, high-delivery absorption candidates.
- Why: the earlier strict profile excluded names like `HINDUNILVR` that showed elevated delivery during weakness because the rule set was still biased toward a stable late-base structure.

### What changed

- Changed all Phase-1 delivery thresholds to `55%` across large, small, micro, and nano buckets.
- Relaxed strict `200 DMA` proximity so downside is allowed; only names more than `+2%` above the `200 DMA` are rejected.
- Added a focused regression test covering these defaults.

### Key decisions and tradeoffs

- Decision: keep tier counts (`tier_60_count_15d`, `tier_70_count_15d`, `tier_80_count_15d`) as context on the UI instead of using higher cap-specific delivery thresholds.
- Tradeoff: this broadens Phase-1 intake, so more names can pass the first screen and later refinement will matter more.
- Decision: keep the upside cap near the `200 DMA`, but stop penalizing names already trading below it.

- Result: passed

## UI filterable results

- Feature: add column-level filtering for the Wyckoff Phase-1 result table.
- Why: once the scanner returns a wider markdown-friendly set, the user needs quick narrowing on the UI before exporting a working shortlist.

### What changed

- Added Ant Table built-in header filter dropdowns for each visible result column in `WyckoffPhase1Page`.
- Used checkbox-style value selection from the current result set, with Ant’s built-in search inside the filter popup.
- Applied filtering through Ant Table and reused its current filtered row set.
- Added a filtered row count and a `Clear Filters` action.
- Updated `Export JSON` to export only the filtered rows, not the full raw result set.

### Key decisions and tradeoffs

- Decision: use Ant Table’s built-in filter UX instead of a separate external filter strip.
- Why: lower UI clutter and better alignment with table-native behavior.
- Decision: use native value-list filtering instead of custom operator parsing.
- Tradeoff: filtering now matches exact values from the result set, which is simpler and closer to the Ant interaction model shown in the screenshot.

## Delivery reconciliation failure diagnostics

- Feature: make delivery backfill failures explain themselves.
- Why: the cron job was failing with `failed=3` even though the user needed to know which dates failed and why, and the count could be misread as symbol-level failures.

### What changed

- Added typed failed-date tracking in `DeliveryReconciliationJob`.
- Updated the backfill summary log to include failed dates and reasons.
- Updated the threshold exception to include the same failed-date details.
- Kept `source unavailable` dates separate from true failures.
- Added a focused cron-job unit test for the failure-summary formatter.

### Key decisions and tradeoffs

- Decision: fix observability first instead of introducing an exchange-holiday calendar in the same change.
- Why: the immediate blocker was not knowing what the 3 failures were.
- Tradeoff: weekday holiday dates can still appear as candidates and be skipped as unavailable, but true failures are now explicit.

## Delivery unresolved threshold rule

- Feature: change delivery backfill failure logic from failed-date counting to per-date unresolved-symbol quality control.
- Why: holiday/no-file dates should just skip, while real data-quality failure should happen only when a single date has too many unresolved symbols.

### What changed

- Stopped treating every unresolved-symbol date as an immediate reconciliation failure.
- Returned unresolved symbols from `DeliveryReconciliationService` while still writing resolved rows.
- Changed delivery report tolerance from `<1% unresolved` to `<= 5 unresolved symbols` on a date.
- Simplified backfill processing in `DeliveryReconciliationJob` to sequential flow so it stops on the first fatal date.
- Added explicit fail-fast checks when unresolved symbols exceed `5` on any backfill date or final reconciliation date.

### Key decisions and tradeoffs

- Decision: replace batch/parallel backfill with sequential processing.
- Why: simpler reasoning and true stop-on-first-fatal-date behavior fit this single-user tool better.
- Tradeoff: backfill may run a bit slower, but failure behavior is easier to trust and debug.
