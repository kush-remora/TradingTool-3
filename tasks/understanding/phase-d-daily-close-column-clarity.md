The reported `Prev Close / Vol` issue is a UI-semantics bug, not a fresh-field calculation bug. The Phase D backend refresh path intentionally stores `closePrice` as the latest completed daily close, `pctChange` as the day-over-day change from the previous close, `volume` as the latest completed daily volume, and `previousDayVolume` separately for `T-1` wake-up comparison. That behavior is explicit in `PhaseCFreshFieldCalculator` and `PhaseCWatchlistService`.

The actual confusion comes from the frontend column title and unlabeled cell values on `PhaseDScannerPage`. The earlier column said `Prev Close / Vol`, which implied prior close and possibly live volume, while it rendered latest daily close, daily % change, and latest daily volume. The correct fix is frontend-only: keep the truthful `Daily Close / Day Vol` column, and add a separate `T-1 Close / T-1 Vol` column derived from existing fields. `T-1 Vol` comes directly from `previousDayVolume`, and `T-1 Close` can be derived from `closePrice` and `pctChange`, so no SQL or schema change is needed.
